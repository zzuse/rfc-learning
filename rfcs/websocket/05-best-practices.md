# 05. 最佳实践 —— 复杂地形下的铁道指南 (RFC 7936 及拓扑实践)

理论上的 WebSocket 是完美的：只需建立一次连接，货物就能在两端双向、实时地高速流通。但在真实的互联网世界中，从你的浏览器到目标服务器，中间充满了崎岖的“复杂地形”：透明代理、反向代理、负载均衡器（Load Balancers）、应用防火墙（WAF）以及各种网关设备。

**RFC 7936 (Clarifying HTTP/1.1 WebSocket usage) 以及相关的业界最佳实践**，为我们提供了一份避坑指南。

---

## 1. 穿越迷雾：为什么 `ws://` 经常会断开？

在早期的网络环境中，很多企业防火墙或透明代理（Transparent Proxies）都被设计成只认识传统的 HTTP 协议（客运列车）。

当你使用非加密的 `ws://` 时：
1. 客户端发起了 HTTP Upgrade 请求。
2. 某些固执的代理服务器一看：“好家伙，请求已经结束了，但底层的 TCP 怎么还在跑奇怪的二进制数据（WebSocket Frame）？”
3. 代理服务器会认为这是一个畸形的 HTTP 连接，为了安全，**直接强行切断 TCP 铁轨**（Drop connection）。

### 最佳实践 1：永远使用 `wss://` (WebSocket Secure)

**解决方案**：永远使用 `wss://`（基于 TLS 加密的 WebSocket）。
- 为什么有用？当使用 TLS 加密时，整个 HTTP 升级握手和后续的数据帧都被加密成了毫无规律的密文流。
- 中间的代理服务器（除非是解密代理）根本看不出你在跑 WebSocket，它只能看到一条 HTTPS 隧道，因而会选择原样放行。这就好比给你的货运专列盖上了一层隐形斗篷。

---

## 2. 对抗“幽灵断线”：保活机制（Keep-Alive）

很多负载均衡器（比如 Nginx, AWS ALB, HAProxy）都有一个硬性规定：**如果一条 TCP 连接在一段时间内（比如 60 秒）没有任何数据流通，就判定它是“死连接”并将其踢除。**

如果你的 WebSocket 专列刚好在这 60 秒内没有业务数据传输，连接就会被莫名其妙地断开。而最糟糕的是，由于是中间设备悄悄掐断的，客户端和服务器甚至都不知道连接已经死了，这就叫“半开连接（Half-open connection）”。

### 最佳实践 2：实现应用层心跳 (Ping/Pong)

虽然我们之前提到了 RFC 6455 定义了底层的 Ping/Pong 控制帧，但在实际的浏览器 API（原生 `WebSocket` 对象）中，**开发者是没有权限手动发送 Ping 帧的**。浏览器的底层实现是否会自动发 Ping 也因环境而异。

**解决方案**：在应用层自己实现心跳！
1. **客户端定时心跳**：客户端每隔 30 秒发送一个简单的文本消息，如 `{"type": "ping"}`。
2. **服务器响应**：服务器收到后，回复 `{"type": "pong"}`。
3. **断线重连**：如果客户端在发送 ping 之后 5 秒内没收到 pong，就主动执行 `socket.close()` 并立刻进入**断线重连（Reconnect）**逻辑。

这是目前保证 WebSocket 在复杂网络拓扑中长治久安的最有效手段。

---

## 3. 子协议协商（Subprotocols）：不讲暗语不开门

在 HTTP 升级阶段，除了 `Upgrade: websocket`，RFC 规范还支持一个极其重要的头部：`Sec-WebSocket-Protocol`。

这也是 **RFC 7936** 重点规范和澄清的领域：即 IANA 子协议注册表的使用。

因为 WebSocket 只是一个底层的搬运工（货运专列），它并不规定货物的格式。货物是 JSON、XML 还是 Protobuf？服务器怎么知道？

### 最佳实践 3：明确你的应用层协议

客户端在握手时应该告知服务器它想用的“应用语言”（子协议）：
```http
Sec-WebSocket-Protocol: mqtt, wamp, my-custom-chat-v1
```
服务器收到后，从列表中挑一个自己支持的，在响应中回复：
```http
Sec-WebSocket-Protocol: mqtt
```
这样，双方在开始发货前，就明确了：“我们将在这条 WebSocket 铁轨上，使用 MQTT 协议格式来封装货物。” 这种良好的契约精神，是构建大型分布式系统的基石。

---

## 4. 优雅降级与限流控制

1. **连接数风暴**：长连接是非常消耗服务器句柄和内存的。如果遭受攻击，大量的 WebSocket 握手可能导致服务器瘫痪。
   - **应对**：在负载均衡层限制每个 IP 的并发 WS 连接数；对于异常断开的客户端，重连必须采用**指数退避（Exponential Backoff）**策略（比如等 1s, 2s, 4s, 8s 再重连），防止“雪崩效应”压垮刚重启的服务器。

2. **消息积压（背压问题, Backpressure）**：如果发送方的速度远大于接收方处理的速度。
   - **应对**：监控底层的 `bufferedAmount` 属性。如果发现堆积过大，应暂停发送数据，等待缓冲区排空。

经过各种复杂地形的考验，我们的 WebSocket 专列已经非常稳健了。但是，随着 Web 技术的演进，我们迎来了 HTTP/2 高铁时代。在 HTTP/2 时代，为每一个 WebSocket 申请一条独立的 TCP 铁轨显得太奢侈了。
